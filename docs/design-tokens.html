2 · Typography

Two families: Figtree (--font-brand, display/wordmark only) and Geist (--font-ui, everything else). Weights are non-standard numeric axis values (420/520/620/680), exploiting the variable fonts.

3 · Spacing scale

px-named tokens: 2px steps to 16, 4px steps to 32, coarser above. Rule (per header comment): all padding/margin/gap must come from this scale; 1px hairlines stay literal.

4 · Radii & borders

Philosophy: controls are rounded, structural panels are square (--radius-panel: 0), overlays are the rounded exception. Borders are all 1px; hairlines use --line / --border, softer overlay borders mix --line 70% with transparent.

5 · Shadows / elevation

Six elevation tokens. Outer shadows for controls/panels/overlays; two inset shadows for pressed states; one inset ring for armed tools.

7 · Form controls

Focus ring across all controls is ring-2 ring-ring ring-offset-2 → rendered as the 4px double-ring in the "focus" examples.

8 · Overlays & menus

Two distinct overlay treatments coexist: menus/select use rounded-sm (8px) + --shadow-panel + solid --line border; popovers/dialogs/toasts use --radius-overlay (12px) + --shadow-overlay + softened border. Tooltips invert to a dark ink surface.

9 · Chips, badges & status

No pills for content chrome · badges are square (2px). The one round shape is the save-status dot. Status colors: petrol = saved/saving, amber = attention (backup stuck, local safe), red = error.

Observations · inconsistencies noted while extracting

Neutral audit notes. Each cites the token / value so it can be verified in source.

  • Inverted Tailwind radius aliases. @theme inline maps --radius-xl → --radius-panel = 0, while --radius-lg = 10px and --radius-sm/md = 8px. So the nominally "largest" Tailwind radius resolves to a square corner · any rounded-xl usage renders sharp, the opposite of intent.
  • Five near-identical light grays. --surface 0.975, --secondary 0.968, --muted-surface 0.965, --accent 0.955, --track 0.945, --surface-strong 0.925 all sit within a 0.05 lightness band at chroma ≤ 0.013. Several are visually indistinguishable; candidates to merge.
  • Two radius tokens with the same value. --radius-fill: 10px and --radius-track: 10px are identical; --radius-sm and --radius-md both resolve to --radius-control (8px). Redundant handles for the same corner.
  • Font-weight tokens bypassed by literals. The scale is --weight-regular/medium/semibold/strong (420/520/620/680), but .app-name, .rail-brand and the page wordmark hardcode font-weight: 700 · a value not in the scale.
  • Off-scale one-off px values that bypass the spacing/size system. Control heights 30px, 34px, 38px, 26px, 18px, 15px (rail-issue-count line-height, switch geometry, section triggers) are literals outside the --space-* scale. Some are load-bearing (the 34×20 switch math is commented), but they're invisible to the token system.
  • Switch base radius contradicts its rendered shape. Switch (switch.tsx) base class sets rounded-sm (8px square-ish), but every real instance (.settings-switch, .unit-switch) overrides to border-radius: 999px (pill). The base radius is dead · and a pill switch is the one pill in an otherwise no-pill system.
  • Radix radius classes don't match the CSS they pair with. Menu/select items use Tailwind rounded-lg in the .tsx, but the co-located .select-item/.dropdown-menu-item CSS sets --radius-sm; likewise SelectContent uses rounded-sm in JSX while .select-content CSS re-declares the same border/radius. Duplicated (and occasionally divergent) styling across the class-utility and CSS layers.
  • Legacy --radius: 0.25rem (4px) is orphaned. Kept in :root but the checkbox that would use a 4px corner hardcodes border-radius: 4px instead of referencing it; nothing consumes --radius directly.
  • Two overlay border treatments. Popover/dialog soften the border via color-mix(in oklch, --line 70%, transparent); menus/select use solid --line. Intentional per comments, but worth confirming it reads as one system.
  • Danger has two "strong" ramps but caution has none. --destructive/--destructive-strong and --primary/--primary-strong both provide hover-darken steps; --caution has only soft/line companions, so caution-tinted interactive states can't darken consistently with the others.
